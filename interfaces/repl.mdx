The REPL (Read-Eval-Print Loop) provides an interactive terminal interface for managing Warden tasks locally. It's started automatically when you run `npm run dev` and offers both direct commands and task queue operations.

## Starting the REPL

<Steps>
  <Step title="Start the development server">
    ```bash
    npm run dev
    ```
    This launches the REPL alongside the task runner and Telegram bot (if configured).
  </Step>
  
  <Step title="Wait for the prompt">
    You'll see:
    ```
    Warden REPL. /new = reset, /context = show context, /tasks = active, /cronJobs = cron, /quit = exit.
    warden>
    ```
  </Step>
</Steps>

## Submitting Tasks

Type any instruction at the prompt to queue a task:

```bash
warden> Check disk space and clean up logs older than 30 days
Task queued: a3f8c2b4-...
```

Tasks are inserted into the Supabase queue with `metadata.source = "repl"` and picked up by the runner.

### Task Inspection

<CodeGroup>
```bash /tasks
# List all active tasks (pending or running)
warden> /tasks
  a3f8c2b4  running    45s  Check disk space and clean up logs
  b7e9d1a2  pending    12s  Update npm dependencies
```

```bash /history
# Show recent tasks (all statuses, default: 3)
warden> /history 5
  c4f2a8e1  done      2m  Deploy to production
  a3f8c2b4  running  45s  Check disk space
  f9b3c7d5  failed    3m  Invalid SSH credentials
```

```bash /activeTask
# Show currently running task with tool invocations
warden> /activeTask
```

<CodeGroup>

```bash /deleteCron

warden> /deleteCron d4a9

```
</CodeGroup>

### Skills

<CodeGroup>
```bash /skills
# List available skill guides
warden> /skills
Available skills: content-style, co-marketing, seo-audit, publish
```

```bash /skill
# Run a task with a skill guide loaded into context
warden> /skill seo-audit Audit homepage for keyword optimization
Task queued (skill: seo-audit): f2c8a4e9-...
```
</CodeGroup>

<Tip>
Skills are reusable workflow guides stored in `skills/*.md`. They provide specialized instructions for content writing, SEO, publishing, and partner monitoring.
</Tip>

### Exit

```bash
warden> /quit
Goodbye.
```

Alternatively, press `Ctrl+C` twice to force exit.

## Source Reference

The REPL implementation lives in `src/repl.ts`:

<CodeGroup>
```typescript Session Reset
// src/repl.ts:42-46
if (input === "/new") {
  markNewSession("repl");
  console.log("Session reset. Starting fresh.");
  rl!.prompt();
  return;
}
```

```typescript Task Queuing
// src/repl.ts:223-228
try {
  const task = await insertTask({ 
    instruction: input, 
    metadata: { source: "repl" } 
  });
  console.log(`Task queued: ${task.id}`);
} catch (err) {
  console.error("Failed to queue task:", err);
}
```
</CodeGroup>

## Session Persistence

The REPL maintains a persistent session stored in `~/.warden/sessions/repl.jsonl`. This means:

- Conversation history survives restarts
- The agent remembers context from previous interactions
- Use `/new` to reset when you want a clean slate

<Note>
Session files are managed by `SessionManager` from `@mariozechner/pi-coding-agent` and stored as JSONL (one JSON message per line).
</Note>

## Architecture Flow

```
User types instruction
        |
        v
  insertTask() → Supabase (status: pending)
        |
        v
  runner.ts polls → claims task → executes
        |
        v
  Result written back to Supabase
```

The REPL doesn't execute tasks directly—it queues them for the runner, just like the Telegram interface.
