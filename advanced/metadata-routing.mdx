Warden uses **task metadata** to route results back to their source (Telegram, REPL, cron jobs) and maintain conversational context. Metadata is a JSONB column that flows through the entire task lifecycle.

## Metadata Flow Architecture

From `~/workspace/source/src/runner.ts:79-116`:

```typescript
async function executeTask(task: Task, session: AgentSession) {
  const key = deriveSessionKey(task.metadata);
  if (key) {
    console.log(`[runner] Session: ${key} (persistent)`);
  }

  // Expose current task's metadata so cron-cli can auto-inherit it
  if (task.metadata) {
    process.env.WARDEN_TASK_METADATA = JSON.stringify(task.metadata);
  }

  try {
    await session.prompt(task.instruction);
    const result = ctx.assistantText || "(no output)";
    await completeTask(task.id, result);
    await notifyTaskComplete({ ...task, status: "done", result });
  } catch (err) {
    await failTask(task.id, msg);
    await notifyTaskComplete({ ...task, status: "failed", error: msg });
  } finally {
    delete process.env.WARDEN_TASK_METADATA;
  }
}
```

<Steps>
  <Step title="Task Created">
    Metadata attached when task is inserted (REPL, Telegram, cron job)
  </Step>
  <Step title="Session Routing">
    `deriveSessionKey()` checks metadata to determine if task belongs to a persistent session
  </Step>
  <Step title="Environment Injection">
    `WARDEN_TASK_METADATA` env var set so agent can create child tasks with inherited metadata
  </Step>
  <Step title="Result Routing">
    `notifyTaskComplete()` reads metadata to send results back (Telegram, etc.)
  </Step>
  <Step title="Cleanup">
    Environment variable cleared after task completes
  </Step>
</Steps>

## Metadata Structure

Common metadata fields:

| Field | Type | Purpose | Example |
|-------|------|---------|----------|
| `source` | string | Origin of task | `"telegram"`, `"repl"`, `"api"` |
| `chatId` | number | Telegram chat ID for replies | `123456789` |
| `cron` | boolean | Task generated by cron job | `true` |
| `sessionKey` | string | Force specific session | `"telegram-123456789"` |

```typescript
export async function notifyTaskComplete(task: Task): Promise<void> {
  if (task.metadata?.source !== "telegram") return;
  if (!task.metadata?.chatId) return;

  const chatId = task.metadata.chatId as number;
  
  if (task.status === "done") {
    await bot.api.sendMessage(chatId, task.result || "(no output)");
  } else if (task.status === "failed") {
    await bot.api.sendMessage(chatId, `❌ Task failed: ${task.error}`);
  }
}
```

<Note>
  All Telegram tasks automatically include `source: "telegram"` and `chatId: <number>` metadata for bidirectional routing.
</Note>

## Session Persistence

### Deriving Session Keys

From `~/workspace/source/src/session-store.ts` (conceptual example):

```typescript
export function deriveSessionKey(metadata: Record<string, unknown> | null): string | null {
  if (!metadata) return null;
  
  // Explicit session key (highest priority)
  if (metadata.sessionKey) return metadata.sessionKey as string;
  
  // Telegram tasks → persistent session per chat
  if (metadata.source === "telegram" && metadata.chatId) {
    return `telegram-${metadata.chatId}`;
  }
  
  // REPL tasks → single persistent session
  if (metadata.source === "repl") {
    return "repl";
  }
  
  // No persistent session (ephemeral task)
  return null;
}
```

### Metadata Inheritance

From `~/workspace/source/src/resolve-metadata.ts:1-25`:

```typescript
/**
 * Resolves task metadata for cron jobs.
 * If explicit JSON string is provided (from --metadata flag), parses and returns it.
 * Otherwise falls back to WARDEN_TASK_METADATA env var (set by runner.ts during task execution).
 * Throws on invalid explicit JSON (caller should handle). Silently ignores malformed env var.
 */
export function resolveTaskMetadata(
  explicitJson: string | undefined
): Record<string, unknown> | undefined {
  if (explicitJson) {
    return JSON.parse(explicitJson);
  }

  const envJson = process.env.WARDEN_TASK_METADATA;
  if (envJson) {
    try {
      return JSON.parse(envJson);
    } catch {
      return undefined;
    }
  }

  return undefined;
}
```

This enables **child tasks to inherit parent metadata**:

```typescript
// Parent task (from Telegram)
const parentTask = {
  instruction: "Schedule a reminder for tomorrow",
  metadata: { source: "telegram", chatId: 123456789 },
};

// Agent creates child cron job → inherits metadata via WARDEN_TASK_METADATA env var
const childJob = await insertCronJob({
  name: "reminder",
  at_time: "2026-03-11T09:00:00Z",
  instruction: "Remind user about meeting",
  task_metadata: resolveTaskMetadata(undefined), // Reads from env → inherits parent metadata
});

// When child job fires → result routes back to original Telegram chat
```

<Warning>
  `WARDEN_TASK_METADATA` is set during task execution and cleared afterward. Only available within the agent's tool execution context.
</Warning>

## Metadata in Database

From `~/workspace/source/supabase/migrations/002_add_task_metadata.sql` (conceptual):

```sql
-- Add metadata column to tasks
ALTER TABLE warden_tasks
ADD COLUMN metadata jsonb;

-- Index for filtering cron tasks
CREATE INDEX idx_warden_tasks_metadata_cron
  ON warden_tasks ((metadata->>'cron'))
  WHERE metadata->>'cron' = 'true';
```

Query examples:

```sql
-- All Telegram tasks
SELECT * FROM warden_tasks
WHERE metadata->>'source' = 'telegram'
ORDER BY created_at DESC;

-- Tasks for specific Telegram chat
SELECT * FROM warden_tasks
WHERE metadata->>'source' = 'telegram'
  AND (metadata->>'chatId')::bigint = 123456789
ORDER BY created_at DESC;

-- All cron-generated tasks
SELECT * FROM warden_tasks
WHERE metadata->>'cron' = 'true'
ORDER BY created_at DESC;

-- Failed cron tasks
SELECT * FROM warden_tasks
WHERE metadata->>'cron' = 'true'
  AND status = 'failed'
ORDER BY created_at DESC;
```

## Best Practices

<Accordion title="Metadata Design Guidelines">
  1. **Always include `source`** — Makes filtering and routing explicit
  2. **Use typed fields** — Keep chatId as number, not string
  3. **Avoid sensitive data** — Metadata is stored in plaintext JSONB
  4. **Document custom fields** — Add comments for team-specific metadata
  5. **Test inheritance** — Verify child tasks route correctly
</Accordion>

<Accordion title="Session Key Conventions">
  - `telegram-<chatId>` — Telegram chat sessions
  - `repl` — Interactive REPL session
  - `api-<userId>` — API-triggered tasks (if implementing external API)
  - `webhook-<sourceId>` — Webhook-triggered tasks
</Accordion>
